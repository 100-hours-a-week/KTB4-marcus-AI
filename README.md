## 1. 엔터키 인식

```python  
for i in range(len(sentences)):  
    revise = sentences[i].rstrip() + ' \n'  
```  
본래 `revise = sentences[i][:-1] + ' \n'`로 해결하려고 했으나,  
{LLM} 내 기대와 달리 마크다운의 문법은 마지막이 엔터로 끝나지 않을 수도 있었다  
그래서 `rstrip()` 메소드를 이용하여 끝부분의 공백이나 엔터키를 없애고,  
우리의 목표인 스페이스바 두번 + 엔터키로 만든다  

`.readlines()` 문법으로 읽어들인 문장들에 대해서 수행한다  

# ~~2. \[\[]] 형식 변경~~  

>옵시디언은 위키랑 같이 \[\[다른 파일 이름]]의 형식으로 다른 파일로 이동할 수 있다.  
>이는 표준적인 마크다운에서 쓰는 문법이 아니므로, 괄호를 지운 안의 글자만 남긴다.  

>\[\[다른 파일 이름 | 원하는 문구]]의 형식의 옵시디언의 문법도 있는데,  
>이 경우 괄호와 '다른 파일 이름 |' 부분까지 삭제한다  

```python  
    while(revise.count('[[') > 0 or revise.count(']]') > 0):  
        a = revise.find('[[')  
        b = -1  
        if '|' in revise[a:c]:  
            b = revise[a:c].find('|') + a  
        c = revise.find(']]')  
        
        if a < b and b < c:  
            revise = revise[:a] + revise[b:c] + revise[c+2]  
        else:  
            revise = revise[:a] + revise[c+2:]  
```  

이런 식으로 왼쪽부터 `'[['`나 `']]'` 등을 찾아서  
slicing한 것들을 더하는 것을 while문으로 접근한 것은 좋은데,  
조금 더 구체적으로 마크다운 변환 규칙을 적용하여 CLI를 만들 경우,  
**다음의 규칙이 발목을 잡는다.**  

## 규칙1)  

>**ㄱ. '\['은 때에 따라 죽여야 한다**:  
>마크다운에서는 뒤에 '( )'괄호랑 합쳐져 url 역할을 하므로 살려야 한다.  
>그러나 '( )' 없이는 독자적으로 마크다운에서 작동하지 않으나,  
>github의 경우 해당 '\[ ]' 괄호가 참조 링크로 잘못 사용될 위험성이 있어서 지워야 한다  
>**ㄴ. '\\\['은 살려야 한다**:  
>url로 작동하지 않는 의도된 표기니 '\['를 원문대로 냅두고, 이에 대응하는 ']'도 살린다  

## 규칙2)  

>**ㄱ. '\[\['은 죽여야 한다**:  
>마크다운에서는 지원하지 않는 위키링크를 사용한다  
>**ㄴ. '\\\[\['은 '\['처럼 죽여야 한다**:  
>위키링크로 작동하지 않는 의도된 표기니 '\['를 원문대로 냅두고, 가장 뒤의 ']'도 살린다  
>그러나 안 쪽의 '\[ \]' 괄호는 첫 번째 규칙을 적용한다  

## 규칙3)  

>**ㄱ. '\!\[\['은 죽여야 한다**:  
>마크다운과는 다른 방식의 사진 불러오는 문법이니 '\!\[\['랑 ']]'랑 그 사이의 것을 다 지운다  
>**ㄴ. '\\!\[\['은 '\[\['처럼 죽여야 한다**:  
>사진을 불러오지 않는 의도된 표기이므로 '!'는 살린다  
>그러나 뒤의 '\[\['와 이에 대응하는 ']]'은 두 번째 규칙을 제거한다  

이 규칙들을 생각하면 **for문 정도로는 쉽게 해결되지 않음을 알 수 있다**  
'\['을 찾는 순간 위의 6개의 기호들이 전부 인식될 것이다  

 
# 2. 괄호 기호의 정규식 문법 방식 사용  

****정규식**: 텍스트에서 특정 패턴을 찾기 위한 검색 명령어**  

그래서 필자는 `import re`와 정규식 문법/원시 문자열 방식으로 문제를 해결하고자 했다  
정규식 문법을 통해 상기된 복잡한 규칙들을 표현하는 문자열을 조건으로써 입력할 수 있고,  
`import re`를 통해 정규식 전용의 replace 문법을 구사할 수 있다  


## 2.1 '\!\[\['은 죽여야 한다  

**ㄱ. 앞에 \가 무조건 없어야 한다** `(?<!\\)`  
- `(?<!조건)`: 글자 앞에 조건이 없어야 한다  
정규식은 `\`를 명령어로 인식하므로, `\\`를 써서 `\`를 쓴다  
이걸 안 하면, 살려야 할 기호도 삭제된다  

**ㄴ. 다음에 !\[\[가 나와야 한다** `!\[\[`  
- `\[`: `[` 자체도 정규식에서 의미를 가지므로, `\[`로 표기해야 한다  

**ㄷ. 사이에 글자가 있다** `.*?`  
- `.` : 아무 글자 한 글자  
- `*`: 앞의 글자가 0번 이상 반복  
다만 이렇게 `.*`를 썼다가는 `![[사진1]] 그리고 ![[사진2]]에서 ...` 같은  
예시에서 앞의 `![[`와 뒤의 `]]`가 전부 인식된다  
- `?`: `?`는 본래 앞의 글자가 0번 또는 1번 있음을 뜻하지만,  
다른 의미로 `*` 같은 기호 앞에 있으면 **lazy하게** 맨 처음 만나는 `]]`에서 멈춘다  

**ㄹ. 다음에 ]]가 나와야 한다** `\]\]`  
- `\]`: 역시 ]가 정규식에서 의미가 있으므로 `\]`를 사용한다  

```python  
pattern1 = r"(?<!\\)!\[\[.*?\]\]"  
# ...  
revise = re.sub(pattern1, "", revise)  
```  
`revise`에서 `pattern1`의 조건에 해당하는 부분을 전부 `""`로 없앤다  

## 2.2 '\\!\[\['은 '\[\['처럼 죽여야 한다  

!\[\[기호는 삭제되었다!  
그러니 \\!\[\[ 기호가 보인다면 앞의 \\만 지우고, 뒤의 '\[\['는 다른 방식으로 없애는게 옳다  
옵시디언에서는 !에 \\를 붙이지만, 표준적인 마크다운 문법에서는 남기 때문이다  
후에 '\[\['를 삭제할 때 없어질 것이므로 \\만 지우자  

**ㄱ. \\!\[\[내용\]\]이 나와야 한다, 대신 저장을 곁들인** `\\(!\[\[.*?\]\])`  
- `(저장내용)`: \[\[내용]] 꼴은 2단계 규칙에서 같이 사라질거니까 저장시켜야 한다  

```python  
pattern2 = r"\\(!\[\[.*?\]\])"  
# ...  
revise = re.sub(pattern2, r"\1", revise)  
```  
괄호 ( )로 남긴 원시 문자열을 `\1`로 저장할 수 있다  
최종적으로 \\를 뺀 !\[\[내용]]이 남는다  

## 2.3 '\[\['은 죽여야 한다  

**ㄱ. 앞에 \가 무조건 없어야 한다** `(?<!\\)`  

**ㄴ. 다음에 \[\[가 나와야 한다** `\[\[`  

**ㄷ. 사이에 글자가 있다** `.*?`  

**1ㄹ. 다음에 |랑 저장해야 할 글자가 나오고, \]\]가 나와야 한다** `|(.*?)\]\]`  
\[\[파일명|글자]]라면 '글자' 부분이 작성자가 의도한 글이다  
글자 부분을 저장해주자  

**2ㄹ. 다음에 \]\]가 나와야 한다** `\]\]`  
\[\[파일명]]이라면 '파일명' 부분이 작성자가 의도한 글이다  
ㄷ의 `.*?`에 괄호 ( )를 쳐, 파일명 부분을 저장해주자  

그러므로 이 경우 1ㄹ의 pattern을 먼저 적용하고, 2ㄹ의 pattern을 작용해야  
두 경우 모두 다룰 수 있다  

```python  
pattern3 = r"(?<!\\)\[\[.*?\|(.*?)\]\]"  
pattern4 = r"(?<!\\)\[\[(.*?)\]\]"  
# ...  
revise = re.sub(pattern3, r"\1", revise)  
revise = re.sub(pattern4, r"\1", revise)  
```  

## 2.4 '\\\[\['는 '\['처럼 죽여야 하므로 일단 그대로 유지한다  

어차피 \\\[\[]]의 꼴은 안쪽 괄호를 지우고, 백슬래시를 없애면  
github 참조링크처럼 인식되어 의도한 대로 작동하지 않을 수 있으므로,  
이 상태 그대로 유지하는 것이 옳다  

## 2.5 '\['은 죽여야 한다  

**1ㄱ. 앞에 \[가 무조건 있어야 한다** `(?<=\[)`  
- `(?<=조건)`: 글자 앞에 조건이 있어야 한다  

**2ㄱ. 앞에 \가 무조건 없어야 한다** `(?<!\\)`  

**ㄴ. 다음에 \[가 나와야 한다** `\[`  

**ㄷ. 사이에 저장해야 할 글자가 있다** `(.*?)`  

**1ㄹ. 다음에 \]\]가 나와야 한다** `\](?=\])`  
- `(?=조건)`: 글자 뒤에 조건이 있어야 한다  
남기고 싶은 문자는 ] 하나이므로, 두번째 ]은 조건으로 붙인다  

**2ㄹ. 다음에 \]가 나오고, (url주소)가 나오면 안 된다** `\](?!\()`  
- `(?!조건)`: 글자 뒤에 조건이 없어야 한다  
여러 변수가 있겠지만, 웬만하면 ( 하나만 있어도 유지하는게 좋다  

그러므로 이 경우 1ㄴ, 1ㄹ의 pattern을 먼저 적용하고, 2ㄴ, 2ㄹ의 pattern을 작용해야  
두 경우 모두 다룰 수 있다  

```python  
pattern5 = r"(?<=\[)\[(.*?)\](?=\])"  
pattern6 = r"(?<!\\)\[(.*?)\](?!\()"  
# ...  
revise = re.sub(pattern5, r"\1", revise)  
if not re.match(pattern10, revise):  
	revise = re.sub(pattern6, r"\1", revise)  
```  
다음에 (url주소)가 나오는 조건은 기존 마크다운 문법과 같으므로 유지한다  

다만, 후에 말할 \[!caution] 등의 콜아웃 문법을 위해서  
pattern10의 꼴이 아닌 경우에만 \[] 표시를 삭제한다  

## 2.6 '\\\['은 살려야 한다  

\\가 붙어있는 한 참조 링크로 작동될 일은 없으므로, 이대로 유지한다  


# 3.  강조 문법 \== 다루기  

>옵시디언에서는 \==을 사용하여 특정 문자들을 형광펜 표시를 할 수 있으나,  
>이는 표준 마크다운 문법에서 사용되지 않는다.  

>그러나 표준 마크다운 문법에서 \==이 사용되는 경우가 있는데,  
>바로 기본 문장 + (엔터키) + \==으로 헤더 표시를 만드는 경우다.  

```python  
pattern7 = r"^={2,}\s*$"  
```  
우선 헤더 표시로 사용되는 \==키를 예외 처리 하자  

**ㄱ. = 표시가 시작부터 있어야 한다** `^=`  
`^`: 문장의 맨 시작점을 의미한다  

**ㄴ. = 표시가 2개 이상 있어야 한다** `{2,}`  
`{숫자}`: 총 몇 개 있는지를 나타낸다. `,`를 사용하면 이상이나 이하를 나타낸다  

**ㄷ. 뒤에는 아무 것도 없거나 공백만 있어야 한다** `\s*$`  
`\s`: 공백을 의미한다  
`$`: 문장의 맨 끝점을 의미한다  

```python  
pattern9 = r"(?<!\\)==(\S.*?\S|\S)=="  
```  
그리고 문장에 있는 \==키를 볼드체 \*\*로 바꾸도록 하자  

**ㄱ. 앞에 \\가 무조건 없어야 한다** `(?<!\\)`  

**ㄴ. 뒤에 \==이 와야 한다** `==`  

**ㄷ. 문자 안의 \== 바로 옆에 공백이 있으면 안 된다** `==(\S.*?\S|\S)`  
`\S`: 공백을 제외한 모든 문자를 말한다. .\*? 앞 뒤에 놓아서 공백을 막는다  
`|`: or의 역할을 한다. 한 글자만 오는 경우를 대비한다  

**ㄹ. 뒤에 \==이 와야 한다** `==`  

```python  
pattern8 = r"(?<!\\)==((?:(?!\*\*).)*?)=="  
```  
그런데 \== 안에 \*\* 꼴이 있다면 마크다운이 꼬일 가능성이 있다!  
이런 경우는 \== 부분을 아예 삭제시키자  

**ㄱ. 앞에 \\가 무조건 없어야 한다** `(?<!\\)`  

**ㄴ. 뒤에 \==이 와야 한다** `==`  

**ㄷ. \== 안에 \*\*가 있으면 안 된다** `((?:(?!\*\*).)*?)`  
`(?:조건.)`: for문 같은 역할이다.  
`.`을 통해 step 1씩 늘어나며 `(?1\*\*)`를 통해 뒤에 \*\*가 오는지 보고,  
이를 `*?`만큼 반복해라  

**ㄹ. 뒤에 \==이 와야 한다 `==`  

```python  
if not re.match(pattern7, revise):  
    if re.search(pattern9, revise):)  
        revise = re.sub(pattern8, r"\1", revise)  
        revise = re.sub(pattern9, r"**\1**", revise)  
```  
`re.match`는 처음부터 살펴보고,`re.search`는 중간도 살펴본다  
헤더로 쓰이는 \==이 아니면 통과시켜주고,  
\==의 옆에 공백이 없다면 통과시켜주고,  
안에 \*\* 꼴이 있다면 \==을 공백으로 만들고,  
없다면 \==을 \*\* 꼴로 만든다  

# 4. 콜아웃 문법 처리  

콜아웃은 >의 인용구 문법에 \[!단어] 꼴을 붙여 추가적으로 강조할 수 있는 문법이다  
본래 옵시디언에 있는 문법이였으나, github에서도 도입하기 시작했다  
그러나 github의 경우는 단어 부분이 대문자, 옵시디언은 소문자이므로 이를 수정해야 한다  

```python  
pattern10 = r"^(>\s?\[\!)(.*?)(\])"  
```  
그룹을 세 개로 만드는 방식을 이용할거다  

**ㄱ. 처음에 >가 와야 한다** `^>`  

**ㄴ. 다음에 공백이 0칸 혹은 1칸이 온다** `\s?`  

**ㄷ. 다음에 \[!단어] 꼴이 온다** `\[\!.*?\]`  

여기서 대문자로 만든 `.*?` 부분을 그룹 ( )로 감싸고,  
나머지 부분도 없어지지 않고 나타나야 하므로, ( )로 감싼다  
`(>\s?\[\!)`랑 `(.*?)`랑 `(\])`으로.  

```python  
def github_alert(match):  
	return match.group(1) + match.group(2).upper() + match.group(3)  
# ...  
pattern10 = r"^(>\s?\[\!)(.*?)(\])"  
# ...  
revise = re.sub(pattern10, github_alert, revise)  
```  
{LLM} 웃긴건 이 `re.sub` 자체 안에 match라는 변수가 숨어 있는 것이다!  
그래서 `github_alert`를 호출할 때 match를 따로 붙여주지 않아도 실행된다!  

# 5. 코드블럭 예외처리  

그러나 이렇게 하면, 코드 블럭 내부에서는 허용되는 \[\[ ]] 문법 등도 전부 바뀐다  
그러므로 코드블럭 문법 \`\`\`랑 \`을 예외 처리 해주어야 한다  

```python  
in_code_block = False  
# ...  
	if revise.strip().startswith('```'):  
        in_code_block = not in_code_block  
        sentence_list.append(revise)  
        continue  

    if in_code_block:  
        sentence_list.append(revise)  
        continue  
```  

옵시디언의 경우 \`\`\`python 등으로 코드블럭을 정의할 수도 있으므로,  
`startswith` 문법을 사용하고, boolean 값을 통해 문장 확인 여부를 결정짓는다  
\`\`\`의 경우, 시작과 끝 중 어느 경우더라도 문장이 아니니 pass,  
안에 있을 경우 문법을 확인하지 않으니 pass

```python  
pattern_verify = r"`[^`]+`|[^`]+"  
# ...  
for i in range(len(sentences)):

```python  
def find_inline_code(match):  
    match_block = match.group(0)  

    if match_block.startswith("`"):  
        return match_block  
    else:  
        return revise_grammar(match_block)  
```  
코드 블럭 문법 여부에 따라서 바꾸지 않거나,  
평서문인게 확실해지면 `revise_grammar` 함수로 보내 지금까지 했던 문법 변환을 실행한다  

```python  
def revise_grammar(revise):  
    revise = re.sub(pattern1, "", revise)  
    revise = re.sub(pattern2, r"\1", revise)  
    revise = re.sub(pattern3, r"\1", revise)  
    revise = re.sub(pattern4, r"\1", revise)  
    revise = re.sub(pattern5, r"\1", revise)  
    if not re.match(pattern10, revise):  
        revise = re.sub(pattern6, r"\1", revise)  

    if not re.match(pattern7, revise):  
        if re.search(pattern9, revise):  
            revise = re.sub(pattern8, r"\1", revise)  
            revise = re.sub(pattern9, r"**\1**", revise)  

    revise = re.sub(pattern10, github_alert, revise)  

    return revise  
```
